# Chapter 70 — Geometry pipeline: vertex → tess → geom → mesh shaders

The "front end" of the graphics pipeline takes vertex data from buffers and emits triangles ready for rasterization. It is hardware + programmable shaders + a *lot* of API politics.

## The classic pipeline (DX10/OpenGL 3 era)

```
Vertex Shader (VS)
        │
   Hull Shader (HS) ──┐
        │             │
  Tessellator (FF)    │ tess optional
        │             │
  Domain Shader (DS) ─┘
        │
 Geometry Shader (GS) (optional, slow on AMD)
        │
   Primitive Assembly + Clipping/Culling
        │
       Rasterizer
```

Five programmable stages plus fixed-function tessellation. AMD GPUs *have* hardware for all of these but for performance they're collapsed.

## How AMD actually maps stages — the merged-shader pipeline

Modern AMD HW has **two** dedicated pipeline slots between primitive setup and rasterization:

- **HS (Hull Shader)** stage — runs on the geometry processor.
- **GS (Geometry Shader)** stage — also where vertex shading lands.

The driver/compiler **merges** API stages into these slots:

| API stages used | Mapped to AMD HW |
|---|---|
| VS only | HS=skipped, GS=VS-as-GS ("ESGS" / "NGG" path) |
| VS + HS + DS | HS=HS, GS=DS-as-GS |
| VS + HS + DS + GS | HS=HS, GS=GS (worst case) |
| Mesh shader | special "NGG mesh" path |

This merging is invisible to the application but the driver and compiler do real work to make it happen — see `mesa/src/amd/common/ac_nir_lower_*` and the radv stage merging logic.

## NGG (Next-Generation Geometry) — RDNA's big change

NGG, introduced with Vega and matured in RDNA, replaces the legacy ESGS path. Key idea: the hardware no longer needs a separate "ES" stage in front of GS; vertex shading and primitive culling happen together in NGG-format wavefronts that emit position+attributes for **culled** primitives only.

Wins:

- Primitive culling on-chip before any rasterization work.
- Better wave utilization.
- Native support for **mesh shaders** (a more flexible front-end).

NGG is the only path on RDNA2+. Mesa's `radv` and AMD's PAL both target NGG.

## Mesh shaders

Mesh shaders (DX12/Vulkan VK_EXT_mesh_shader) replace the entire VS/HS/DS/GS chain with two stages:

- **Task shader** (optional): emits work for downstream mesh shaders, can amplify work.
- **Mesh shader**: outputs vertices and primitives directly, with explicit primitive connectivity, in a compute-shader-like programming model.

Why it's a big deal:

- App controls the topology (no API limits like GS amplification cap).
- LOD, culling, and small-triangle reject can be done per-meshlet.
- Better fits modern rendering (Nanite-style virtualized geometry).

AMD has supported mesh shaders since RDNA2 (DXR 1.0 era) via NGG.

## Primitive culling

Several culls happen before rasterization:

1. **Frustum cull** — primitive entirely outside view frustum.
2. **Backface cull** — primitive facing away (CW/CCW depending on state).
3. **Small-primitive cull** — sub-pixel triangles that won't generate any pixels.
4. **Zero-area cull** — degenerate.
5. **Cluster culling** (modern, app-controlled) — entire meshlet cull.

These happen on the geometry processor; rejected primitives never enter the rasterizer. NGG made them efficient by doing them on the same wave that did vertex shading.

## Index/vertex fetch

Vertex inputs come from buffers in VRAM. Two paths:

- **Index buffer** + **vertex buffer**: the geometry processor fetches indices, then issues vertex fetches. Cache-friendly with an indexed cache (PA_PRIM cache).
- **Direct draw**: fetches vertices in order.

Vertex format conversion (e.g., `R8G8B8_UNORM` → `vec4`) happens either in fixed function (legacy) or in the VS (modern, generated by the compiler).

## Tessellation (HS + Tess + DS)

DX11-era feature. Hull shader emits per-patch and per-control-point data and tessellation factors. The tessellator (fixed function) generates new vertices according to the factors. The domain shader runs per generated vertex.

Use cases: terrain, character LOD, smoother surfaces. In practice, tessellation has fallen out of fashion vs mesh shaders + virtual geometry. Most AAA engines now use mesh-shader pipelines.

## Geometry shader — the cautionary tale

GS lets you read a primitive and emit zero or more primitives. It looked great on paper but turned out to be **slow on AMD** (and most vendors): the variable output rate forces serialization. NGG path tries to avoid GS when possible, mapping VS/DS-only chains to non-GS hardware.

Avoid GS if you control the renderer.

## Position-only / streamout

Some setups fetch only position attributes for shadow/depth-only passes. AMD supports a "position-only" wave shape that skips fetching unused attributes. Saves bandwidth and BW-bound stages.

**Streamout** writes vertex shader outputs back to a buffer (transform feedback). RDNA implements via direct memory writes from the geometry pipeline. Performance depends on coalescing.

## Driver responsibilities

The kernel driver doesn't really program the pipeline; it just provides the GPU and dispatches submitted IBs. The **userspace driver** (Mesa/radv, AMD PAL/Vulkan, etc.) does:

- Compile API shaders (NIR → ACO/LLVM → AMDGPU ISA).
- Set up RBE/CB state via context registers.
- Emit vertex/index buffer descriptors.
- Pick wave size (32 vs 64), NGG mode, optimal stage merging.
- Emit PM4 to set context state and submit draws.

Kernel just: validates BOs, schedules IBs, signals fences.

## Performance traps in geometry

| Symptom | Cause | Fix |
|---|---|---|
| Vertex throughput collapses | Many small draws | Batch / instance / indirect draws |
| GS used heavily | Slow on AMD | Refactor without GS or to mesh shaders |
| Excessive vertex re-fetch | Index buffer not localized | Mesh-optimize index ordering |
| Position passes slow | Full vertex shader for shadow | Use position-only path |
| Tessellation hot | Over-tessellation | Cap tess factors |
| Mesh shader hangs | Wave divergence in task→mesh emission | Check group sizes, stride in `OutputVertices` |
